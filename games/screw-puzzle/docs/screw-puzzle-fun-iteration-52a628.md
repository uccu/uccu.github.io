# 拧螺丝游戏趣味性迭代计划

本计划围绕更爽的反馈、更丰富的关卡节奏和更低的卡住挫败，对现有静态 H5 拧螺丝游戏做一次小而完整的趣味性迭代。

## 目标

- **增强爽感反馈**：让拆螺丝、板子释放、满箱交付和通关更有即时反馈。
- **丰富关卡变化**：在不推翻现有 50 关生成器的前提下，让不同阶段的形状、偏移、层级和螺丝密度更有差异。
- **降低卡住挫败**：提供更清晰的可点击提示和保底推进手段，减少玩家不知道下一步点哪里或工具箱等待造成的挫败。

## 当前代码理解

- **入口与 UI**：`index.html` 提供顶部状态栏、舞台、弹窗和底部工具箱面板。
- **样式**：`css/style.css` 目前是压缩样式，已有工具箱、飞入、锁定、弹窗等基础表现。
- **核心玩法**：`js/main.js` 管理 Three.js 渲染、Rapier 物理、点击螺丝、遮挡判断、工具箱收纳和胜负判断。
- **关卡生成**：`js/levels.js` 程序化生成 50 关，支持多种板子形状与塔形结构。
- **校验能力**：`scripts/validate-levels.js` 可校验关卡数据与螺丝位置，`node --check` 可做 JS 语法检查。

## 拟实施范围

### 1. 爽感反馈升级

- **拆螺丝反馈**：成功拆下螺丝时增加短暂粒子/光圈反馈，沿用现有 DOM/Three.js 能力，不引入新依赖。
- **板子状态反馈**：板子进入单螺丝悬挂时给板子或剩余螺丝加轻微高亮；最后释放时增加更明显的下落冲击感。
- **满箱与通关反馈**：工具箱满箱交付时增加更明确的弹跳/闪光；通关弹窗文案可加入本关完成度反馈。

### 2. 关卡变化升级

- **生成参数分段**：保留 50 关结构，但细化中后期 `shape`、`sway`、`multiChance`、`baseW/baseH`、`screwsPerBoard` 的组合。
- **阶段节奏**：前 5 关继续教学稳定；6-20 关增加形状差异；21-40 关增加多片板与摆动；41-50 关增强巨塔特色但避免单纯堆数量。
- **可验证安全线**：调整后必须通过关卡校验，避免螺丝落到板子外或颜色/引用异常。

### 3. 降低卡住挫败

- **可点击提示**：当玩家点错或短时间没有操作时，高亮一个当前可推进的螺丝，优先选择最高层且有可用/可解锁工具箱的颜色。
- **工具箱提示优化**：若对应颜色工具箱锁定，提示同时引导玩家点击底部锁定工具箱，而不仅是报错。
- **轻量保底机制**：当无可用工具箱但存在锁定箱或待补箱时，不立即失败；优先给出明确提示，必要时自动刷新一个可推进颜色的锁定箱。

## 预计改动文件

- **`index.html`**：如需新增轻量 HUD/提示按钮，会添加少量 DOM 节点。
- **`css/style.css`**：补充粒子、脉冲、高亮、满箱与提示相关样式。
- **`js/main.js`**：增加反馈动画、提示选择逻辑、工具箱提示和卡住前缓冲逻辑。
- **`js/levels.js`**：调整关卡生成参数和阶段分布。
- **`README.md`**：如玩法规则变化明显，再同步简短说明。

## 验证计划

- **语法检查**：运行 `node --check js/main.js`、`node --check js/levels.js`、`node --check scripts/validate-levels.js`。
- **关卡校验**：运行 `node scripts/validate-levels.js`。
- **浏览器冒烟**：启动本地静态服务器，检查首页加载、重开、上一关/下一关、拆螺丝、满箱、卡住提示和通关弹窗。
- **试玩关注点**：确认反馈不遮挡操作、提示不破坏解谜感、后期关卡变化明显但不卡死。

## 风险与控制

- **避免大改架构**：本次不拆分 `main.js`，以局部函数和样式增强为主。
- **避免新增依赖**：继续保持静态 H5，无构建步骤、无 CDN、无新增第三方包。
- **避免难度失控**：关卡生成调整后先跑校验，再进行少量人工试玩。
