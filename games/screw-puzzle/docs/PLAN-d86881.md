# 拧螺丝物理游戏项目分析与调整计划

`screw-puzzle` 当前已经不是素材预览或待接入物理的空壳，而是一个可运行的静态 H5 物理小游戏原型。计划需要以现有代码为基准继续迭代，避免再按旧的 `Matter.js` / 3 槽工具箱方案推进。

## 当前项目结构

- **入口页面**：`index.html`
  - 负责页面骨架、顶部关卡信息、舞台容器、底部工具箱面板与弹窗。
  - 通过普通 `<script>` 依次加载本地 `Three.js`、`Rapier 2D`、`js/levels.js`、`js/main.js`。
- **样式文件**：`css/style.css`
  - 负责全局布局、舞台背景、顶部信息栏、工具箱、锁定状态、弹窗和移动端适配。
- **关卡数据**：`js/levels.js`
  - 使用 `980 x 560` 设计空间。
  - 程序化生成 50 关。
  - 支持矩形、方形、菱形、三角形、五边形、六边形、八边形等视觉板子形状。
- **核心逻辑**：`js/main.js`
  - 负责启动流程、渲染、物理世界、关卡切换、板子/螺丝创建、点击判定、遮挡判定、工具箱、胜负判定和资源清理。
- **本地依赖**：`vendor/`
  - `three.min.js`
  - `rapier2d-compat.js`

## 当前实现状态

- **技术栈**：实际使用 `Three.js + Rapier 2D`，不是旧文档中的 `Matter.js`。
- **渲染坐标**：使用正交相机，世界单位等于设计像素，原点在左下角。
- **物理世界**：Rapier 重力为向下；左右墙和天花板阻挡板子，底部无地板，便于板子掉出屏幕。
- **板子状态**：
  - 初始为 `Fixed`。
  - 剩 1 个螺丝时转为 `Dynamic`，并用剩余螺丝创建旋转关节。
  - 最后 1 个螺丝拆除后释放板子自由下落。
  - 板子掉出屏幕或横向越界后标记为清除。
- **碰撞策略**：
  - 板子只和墙体碰撞。
  - 板子之间不互相碰撞。
  - 螺丝不参与碰撞，只作为点击对象和旋转关节锚点。
- **螺丝交互**：
  - 玩家点击可见螺丝拆除。
  - 点击命中使用固定半径容错。
  - 被遮挡螺丝会降低透明度并不可点击。
- **遮挡判定**：
  - 依据板子 `z` 层级判断上层板子。
  - 矩形和方形使用旋转矩形范围判断。
  - 三角形、菱形、五边形、六边形、八边形使用精确多边形点包含判断。
- **工具箱系统**：
  - 当前按关卡中出现的颜色生成工具箱，最多 4 色。
  - 前 2 个默认解锁，保证游戏一开始就可推进；另外2 个锁定作为选配，玩家点击就可解锁。
  - 容量按该颜色场上剩余螺丝数动态设置，最多 6 个；剩 1 个螺丝就显示为 1 槽，避免出现填不满的空槽。
  - 已解锁工具箱装满后被“交付”，随后从场上还需要的颜色中随机选一个填回原槽位；如果没有可选颜色则移除槽位。
  - 解锁锁定槽时，如果预设颜色与其他解锁槽重复或该颜色已无剩余螺丝，会重新随机选一个还需要的颜色。
- **关卡流程**：
  - 支持上一关、下一关、重开。
  - 当前关卡索引保存到 `localStorage`。
  - 所有板子清除后通关；无可推进螺丝时判定卡住。

## 与旧计划不一致的地方

- **物理引擎已变更**：旧计划写 `Matter.js`，代码实际为 `Rapier 2D`。
- **工具箱规则已变更**：旧计划写底部同时显示 3 个、每个 3 槽、满箱补队列；代码实际是按颜色生成最多 4 个，容量按剩余螺丝数动态设置（最多 6）。
- **资源方案已变更**：旧 README 提到 `assets/` 素材目录，但当前仓库主要是程序化图形，没有实际 `assets/` 目录。
- **关卡规模已扩展**：旧计划是第一版流程，当前已有 50 关生成逻辑和关卡导航。
- **遮挡精度已升级**：旧计划只提旋转矩形简化；当前已对非矩形视觉板子使用精确多边形遮挡判定。

## 已完成的本轮决策

- **工具箱玩法方向**：保留当前 4 色动态容量模型，不回退到旧的 3 箱队列固定 3 槽模型。
- **锁定工具箱规则**：前 2 个默认解锁作为主推进槽位；另外 2 个锁定作为选配助手。已解锁槽位交付后会刷新为随机“还需要的颜色”；玩家随时可以解锁锁定槽位获得并行能力，从而降低难度。
- **金币文案处理**：当前没有金币系统，因此去掉金币/花费相关文案，只保留点击解锁。
- **多边形遮挡规则**：已使用与视觉板子一致的多边形局部坐标点集做精确点包含检测。
